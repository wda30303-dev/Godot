這份文件提供給後續開發者與 AI coding agent 使用，用來理解本 Godot 專案目前的設計方向、已完成的原型內容，以及未來可能擴充的系統。修改專案前請先閱讀本文件，避免把遊戲做往不符合原本構想的方向。

- 若要新增或修復 UI 文字，請使用可讀的繁體中文。

## 專案概念

這是一款 Godot 4.6 的 2D 策略戰鬥原型，核心概念是把「猜數字」與「自動出兵攻城」結合。

目前設計方向不是單純的猜謎遊戲，也不是純塔防或純自走戰鬥，而是讓玩家透過猜數字取得戰場資源、推進兵線、佔領節點、升級技能，並逐步解鎖更強單位。

主要元素包含：

- 類似 1A2B / Bulls and Cows 的猜數字系統。
- 玩家與敵方左右對攻的自動戰鬥。
- 雙方城堡與城堡 HP。
- 可被雙方士兵佔領的戰場節點。
- 後續預計加入點數、技能、士氣、防禦建築、菁英單位與王單位。

## 目前已實作的雛形

目前專案已有下列主要檔案與系統：

- `game_root.tscn`：主場景。
- `guess_engine.gd`：產生不重複數字答案，並檢查玩家猜測結果。
- `input_panel.gd`：處理玩家輸入、送出與顯示回饋。
- `game_controller.gd`：統籌猜測結果、出兵計時、UI 數值、能量條、雙方單位與戰場節點。
- `PlayerCastle.gd` / `EnemyCastle.gd`：處理雙方城堡 HP，以及進入攻擊範圍的敵方單位造成的傷害。
- `ControlNode.gd`：處理中立、玩家、敵方三種節點歸屬，以及士兵站在節點中心時的佔領流程。
- `Units/`：包含一般士兵、敵方士兵、菁英、特殊敵人、王單位等相關腳本與場景。

目前玩家猜的是 3 位不重複數字。當結果達到 `3A` 時，視為猜中答案，會清空玩家猜測紀錄、重新產生答案、增加特殊能量，並重置本輪的 A/B/C 統計。

目前玩家與敵方單位會依照計時自動生成，往對方城堡方向移動，遇到敵方單位會停下攻擊。進入對方城堡偵測區時，會對城堡造成傷害。城堡 HP 歸零時，目前會暫停遊戲。

目前戰場中間會建立 5 個控制節點。士兵進入節點中央時會停下並開始佔領；若沒有敵方干擾，佔領進度滿後節點會轉為該方所有。

## 後續規劃方向

以下是目前規劃中的系統，未來可能會調整。實作時請保持彈性，不要過早把數值、門檻或公式寫死。

1. 雙方士兵佔領戰場節點後，節點會自動生成防禦建築物。
2. 玩家每次猜測後，依照 A/B/C 等猜測結果給予點數。
3. 點數可以用來升級技能。
4. 技能總數規劃為 10 種。
5. 技能分類暫定如下：
   - 士兵相關技能 3 種。
   - 防禦塔相關技能 3 種。
   - 猜數字相關技能 3 種。
   - 士氣加成技能 1 種。
6. 每個士兵身上都有士氣值。交戰時，依照「交戰處士兵數量 * 士氣值」比較雙方士氣強度。士氣較高的一方可以往前推動戰線。
7. 累積猜對次數後，會依序解鎖菁英單位與王單位。
8. 猜數字相關技能其中之一會讓 `C` 結果出現。`C` 的功能是把玩家猜測的數字與 10 種技能的等級做比對；若猜測數字與任一技能等級數字相同，則 `C + 1`。
9. 猜題長度會隨單位解鎖進度提升：
   - 初始為 3 位數，達成 `3A` 視為猜中。
   - 解鎖菁英單位後變成 4 位數，達成 `4A` 視為猜中。
   - 解鎖王單位後變成 5 位數，達成 `5A` 視為猜中。
10. 菁英單位與王單位都會有特殊被動，可以影響戰場。若菁英與王單位同時存在，只有王單位被動會生效。

## 設計原則

- 明確區分「目前已實作」與「未來規劃」。
- 未經確認前，不要把未定案的技能名稱、數值、費用、解鎖門檻寫死。
- 優先保持系統模組化，讓猜數字、點數、技能、士氣、單位、節點、防禦建築可以分開調整。
- 新增功能時，盡量做成小步驟，方便測試與回頭修改。
- 保留目前原型能運作的部分，除非使用者明確要求重做或替換。
- 若某個規則會影響很多系統，先確認設計再深入實作。

## 建議系統分工

後續新增功能時，可優先依照下列邊界拆分：

- 猜數字系統：答案長度、A/B/C 判定、輸入驗證、猜中判定、猜測紀錄。
- 獎勵系統：每次猜測得到的點數、猜中後的額外獎勵。
- 技能系統：10 種技能定義、等級、升級花費、解鎖條件、效果套用。
- 單位系統：士氣、移動、攻擊、特殊被動。
- 戰場系統：控制節點、防禦建築、兵線推進、城堡傷害。
- 進度系統：猜中次數、菁英解鎖、王單位解鎖、猜題長度提升。
- UI 系統：輸入面板、結果顯示、點數顯示、技能升級介面、狀態提示。

## 目前需要注意的實作細節

- 現有單位使用 `ally`、`enemy`、`player_units`、`enemy_units` 等 group。新增單位場景時請維持一致，避免城堡偵測、節點佔領或戰鬥判定失效。
- 城堡傷害目前會尋找單位的 `attack_power` 或 `get_attack_power()`。若調整單位攻擊屬性，請同步檢查城堡傷害邏輯。
- 多個單位腳本目前有重複的 HP、移動、攻擊、死亡與數字標籤邏輯。若後續大幅擴充單位種類，可考慮整理成共用基底，但不要在小改動時做過大的重構。
- `ControlNode.gd` 目前會在士兵佔領節點中心時鎖住移動。未來加入士氣推線時，需要考慮這個鎖定行為如何互動。
- `guess_engine.gd` 目前的「不可重複數字」檢查需要重新檢視，尤其未來要擴充到 4A、5A 時更需要修正。
- 目前專案中文字有亂碼。若修改 UI 或註解，建議順手改成可讀繁體中文，但不要為了修文字而改動不相關邏輯。


## 後續工作方式

- 修改前先讀相關場景與腳本，不要只看單一檔案就下判斷。
- 優先完成使用者當下指定的功能，避免順手做太多未要求的重構。
- 若設計還在變動，先用容易替換的資料結構與清楚的 API。
- 新增功能後，盡可能執行 Godot 專案或做針對性的腳本檢查。
- 若無法執行 Godot，也要在回覆中說明哪些部分已檢查、哪些尚未驗證。
